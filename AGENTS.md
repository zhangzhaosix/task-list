# PROJECT AGENTS（强制执行）

## 1. 项目前置要求
执行任何任务前，必须先阅读 `README.md`，理解以下内容：
- 项目用途
- 技术栈
- 目录结构
- 核心模块职责
- 启动、构建、部署方式

未完成前置理解前，禁止：
- 修改代码
- 创建功能文件
- 重构项目
- 开始开发

如果项目缺少 `README.md`，必须先根据当前项目生成基础 `README`，至少包含：
- 项目介绍
- 技术栈
- 本地启动方式
- 环境变量说明
- 常见问题
- 部署注意事项

## 2. 任务执行流程
所有任务默认按以下流程执行，禁止跳步：
需求分析
→ 任务拆分
→ UI 设计（如涉及前端界面）
→ 设计审查
→ 开发实现
→ 自动化验证
→ 问题修复
→ Code Review
→ README 同步检查
→ 收尾说明

开始开发前必须输出：
- 目标
- 验收标准
- 执行计划
- 主要风险点

先理解，后行动；不确定需求时，先复述理解与疑问，经用户确认再动手。

每次改动前，先想最可能出问题的 3 个点。

存在关键不确定性时，必须先向用户确认，禁止猜测后直接实现。

收尾说明必须简洁说明：
- 本次完成内容
- 验证结果
- `README.md` 是否需要更新及原因
- 是否存在无关改动

## 3. 工具使用规则
执行任务前必须声明：
- 准备使用哪些工具
- 每个工具的用途
- 为什么需要使用

如出现工具缺失、权限不足、配置缺失或服务不可用，必须说明：
- 缺失内容
- 影响范围
- 最小替代方案

禁止擅自：
- 安装工具
- 修改环境
- 申请权限
- 切换方案

必须等待用户确认后再继续。

默认使用自然的简体中文输出，避免英文直译腔。

## 4. 工具职责
- `Superpowers`：负责需求分析、任务拆分、技术方案、TDD、调试和 Code Review，禁止直接编码。
- `ui-ux-pro-max`：负责 UI 设计、页面结构和视觉规范。涉及前端页面时，必须先设计再开发。
- `Impeccable`：负责 UI 质量审查。核心页面必须执行 `/impeccable audit` 和 `/impeccable polish`。
- `Codex`：负责功能开发和缺陷修复，要求小步迭代，持续保持项目可运行。
- `Chrome DevTools MCP`：负责自动化测试和功能验收，必须验证页面可访问、核心流程正常、Console 无报错。
- `Ponytail`：负责反过度工程、最小实现、依赖控制和代码简化审查。

涉及较多代码、新依赖、公共组件、工具函数或重构时，完成后必须执行：
@ponytail-review

未通过必要审查和验证前，不得宣布任务完成。

## 5. 编码原则
必须遵守：
- 先思考，后编码
- 最小改动
- 精准修改
- 简洁优先
- 保持可逆性
- 尊重项目现有风格
- 优先使用标准库、浏览器原生能力、框架内置能力和项目已有依赖

禁止：
- 顺手重构
- 顺手优化
- 修改无关模块
- 过度设计
- 过度封装
- 提前开发未来需求
- 无必要新增依赖
- 大规模重构

不得牺牲：
- 安全性
- 校验逻辑
- 错误处理
- 可访问性
- 边界处理

历史问题可以指出，但不得在当前任务中主动修改，除非用户明确要求。

默认不额外生成文档；除非本规则要求或用户明确要求，不主动写 README / 设计文档。

## 6. README 同步规则
出现以下情况，必须同步更新 `README.md`：
- 新增功能
- 修改配置
- 修改启动方式
- 修改部署方式
- 修改环境变量
- 修改核心使用说明

如果本次修改不影响 `README.md`，也必须在收尾时说明：
`README.md` 无需更新，原因：xxx

如果项目缺少 `README.md`，必须先补基础内容，再继续任务。

## 7. 完成标准
只有同时满足以下条件，才允许宣布完成：
- 需求已实现
- 功能已验证
- 自动化测试通过
- Console 无报错
- 无无关改动
- `README.md` 已检查并按需同步
- Code Review 已完成

禁止使用以下表述：
- 应该可以
- 理论可用
- 大概没问题
- 未验证完成

所有完成结论必须基于实际验证结果。

## 8. GitHub 同步与上传安全规则
当用户明确表达以下含义时，默认允许同步 GitHub，无需再次询问：
- 我满意了
- 可以了
- 没问题了
- 验收通过
- 提交吧
- 推送吧

同步前必须完成 Git 安全检查。

### 8.1 Git 安全检查
在执行任何 `git add`、`git commit`、`git push` 之前，必须检查项目根目录是否存在 `.gitignore`。

如果不存在 `.gitignore`，必须先创建。

如果已存在，必须确认已忽略以下内容：
- `.env`
- `.env.*`
- `*.local`
- `node_modules/`
- `dist/`
- `build/`
- `.next/`
- `.nuxt/`
- `.vercel/`
- `.netlify/`
- `.cache/`
- `.turbo/`
- `*.log`
- `logs/`
- `.DS_Store`
- `Thumbs.db`
- `.idea/`
- `.vscode/`
- `*.zip`
- `*.rar`
- `*.7z`
- `*.pem`
- `*.key`
- `credentials.json`
- `temp/`
- `tmp/`

### 8.2 禁止上传内容
禁止上传：
- 密钥、Token、密码
- 证书、私钥
- 依赖目录
- 构建产物
- 缓存文件
- 日志文件
- 系统垃圾文件
- 无关大文件
- 临时测试文件
- 调试截图
- Codex 生成的无关杂物

如果敏感文件已经被 Git 跟踪，不能只写 `.gitignore`，必须停止上传并提醒用户处理，例如：
`git rm --cached <文件路径>`

### 8.3 提交前检查
提交前必须执行：
- `git status --short`
- `git diff --cached --name-only`

禁止为了方便直接 `git add .` 后提交。

必须先查看变更清单，确认提交内容只包含与本次任务相关的文件，且不包含敏感文件、无关文件、依赖目录和构建产物。

### 8.4 默认推送流程
默认流程：
1. 检查 `README.md` 是否需要更新
2. 检查 `.gitignore`
3. 检查 Git 状态和暂存文件
4. 提交本地变更
5. 推送至远程 GitHub 仓库
6. 如果没有特别指定分支，固定推送到 `main`
7. 返回 Commit 信息
